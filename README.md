# IFRS 9 ECL Calculator

[![CI/CD](https://github.com/dillonsnyman1/ifrs9-ecl-calculator/actions/workflows/ci-cd.yml/badge.svg)](https://github.com/dillonsnyman1/ifrs9-ecl-calculator/actions/workflows/ci-cd.yml)

A full-stack demo that classifies a loan portfolio into IFRS 9 Stage 1 / 2 / 3
and calculates Expected Credit Loss (ECL) at loan and portfolio level.

- **Backend**: Python + FastAPI for the staging logic and ECL engine
- **Frontend**: React + Vite + TypeScript dashboard (summary metrics,
  staging/ECL charts, filterable loan table, CSV upload)

> **Disclaimer**: This is a simplified, illustrative implementation built for
> portfolio purposes. It is **not** a production-grade IFRS 9 model and
> should not be used for regulatory reporting. All data is synthetic.

---

## Methodology

Each loan has the following inputs: `product_type` (the portfolio segment
the loan belongs to), `exposure_at_default` (EAD), `lgd`, `pd_12m` (current
12-month PD), `pd_origination` (PD at origination), `days_past_due`,
`remaining_term_months`, and `eir` (effective interest rate, used for
discounting).

In practise ECL is calculated separately for each portfolio (mortgages,
personal loans, SME etc), since each has its own PD/LGD models and
parameter ranges. `product_type` is a free text field for whatever label
your portfolio uses - the bundled sample is a single retail personal loan
portfolio, generated by `app/data_generator.py`.

### Staging

| Stage | Criteria |
|---|---|
| **Stage 3** | `days_past_due >= 90` (credit-impaired / default) |
| **Stage 2** | `days_past_due >= 30` OR `pd_12m / pd_origination >= 2.0` (significant increase in credit risk) |
| **Stage 1** | Otherwise |

The `90`, `30` and `2.0` thresholds above are the defaults, but they're
configurable. The dashboard has controls to set your own SICR PD multiple
and stage 2/3 DPD backstops, so you can see how sensitive the staging (and
therefore the ECL) is to the thresholds chosen. The stage 3 backstop must
be greater than or equal to the stage 2 backstop.

### ECL Calculation

- **Lifetime PD** is approximated by compounding the 12-month PD over the
  remaining term: `pd_lifetime = 1 - (1 - pd_12m) ^ (remaining_term_months / 12)`, capped at 1.0.
- **Stage 1** → 12-month ECL = `pd_12m * lgd * exposure_at_default`
- **Stage 2 / 3** → Lifetime ECL = `pd_lifetime * lgd * exposure_at_default`

### Discounting to Present Value

IFRS 9 requires ECL to be discounted to present value at the loan's
effective interest rate (EIR), for both 12-month and lifetime ECL. This
implementation doesn't model a full cash shortfall schedule, so as a
simplification it assumes the loss crystallises at a single point within
the relevant ECL horizon. Two timing assumptions are supported, selectable
via the `discount_method` parameter (or the dropdown on the dashboard):

- **`midpoint`** (default): the loss crystallises half way through the
  horizon (0.5 years for stage 1, mid-point of the remaining term for
  stage 2/3)
- **`end_of_horizon`**: the loss crystallises at the end of the horizon
  (1 year for stage 1, the full remaining term for stage 2/3). This is
  more conservative, giving a higher PV for the same undiscounted ECL.

`ECL = (PD x LGD x EAD) / (1 + eir) ^ years_to_loss`

Both the discounted and undiscounted ECL figures are shown throughout the
app, so the impact of discounting is visible at a glance. Portfolio-level
results aggregate total exposure, total discounted and undiscounted ECL,
the ECL coverage ratio, and counts/exposure/ECL broken down by stage.

### Macroeconomic Scenarios

IFRS 9 requires ECL to reflect a probability-weighted average across
multiple forward-looking macroeconomic scenarios, not just a single
base-case PD. Scenario weighting is **off by default** - with the toggle
off, ECL is the original single-PD calculation (100% weight on a single
unadjusted scenario). Switching it on lets you model three scenarios -
**base**, **upside** and **downside** - each defined by a probability
`weight` and a `pd_multiplier` applied to a loan's `pd_12m` (the resulting
adjusted PD is capped at 1.0 and then rolled forward into an adjusted
lifetime PD using the same compounding formula as above).

The defaults are:

| Scenario | Weight | PD multiplier |
|---|---|---|
| Base | 60% | x1.0 |
| Upside | 20% | x0.8 |
| Downside | 20% | x1.5 |

Weights must sum to 100%. For each loan, ECL is calculated once per scenario
(using that scenario's adjusted 12m/lifetime PD) and then combined into a
single probability-weighted `ecl` / `ecl_undiscounted` figure. The per-scenario
figures are also returned (`ecl_scenarios` / `ecl_undiscounted_scenarios` per
loan, and `by_scenario` at portfolio level), so you can see what ECL would be
if a single scenario played out with certainty versus the blended figure the
dashboard highlights.

**Staging basis**: by default (`base_case`), the Stage 1/2/3 classification
(including the SICR PD-ratio test) uses the loan's own `pd_12m` /
`pd_origination`, unaffected by the scenario multipliers - DPD backstops are
always scenario-independent. Switching to `scenario_weighted` instead uses
the probability-weighted PD across all three scenarios for the SICR ratio
test, which can move borderline loans from Stage 1 to Stage 2. Either way, a
single stage is determined per loan and used consistently across all three
scenarios' ECL calculations.

## Running Locally

### Backend

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

The API runs at `http://localhost:8000`.

Run the test suite:

```bash
pytest
```

### Frontend

```bash
cd frontend
npm install
npm run dev
```

The dashboard runs at `http://localhost:5173` and expects the backend API at
`http://localhost:8000`.

## API Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/portfolio` | Returns the bundled synthetic sample portfolio, processed (staging + ECL) |
| `POST` | `/api/portfolio/upload` | Accepts a CSV upload of a custom loan portfolio, returns processed results |
| `GET` | `/api/portfolio/sample-csv` | Downloads the sample portfolio CSV as a template for custom uploads |

Both `/api/portfolio` and `/api/portfolio/upload` accept an optional
`discount_method` query parameter (`midpoint` or `end_of_horizon`,
default `midpoint`) to control the discounting timing assumption described
above.

They also accept optional query parameters to override the staging
backstops:

| Parameter | Default | Description |
|---|---|---|
| `sicr_pd_multiple` | `2.0` | Loan moves to stage 2 if `pd_12m / pd_origination` is at least this multiple |
| `stage_2_dpd_threshold` | `30` | Days past due backstop for stage 2 |
| `stage_3_dpd_threshold` | `90` | Days past due backstop for stage 3 (default) |

`stage_3_dpd_threshold` must be greater than or equal to
`stage_2_dpd_threshold`, otherwise the API returns a 422.

They also accept optional query parameters to configure the macroeconomic
scenarios used for probability-weighted ECL (see Methodology above):

| Parameter | Default | Description |
|---|---|---|
| `scenario_base_weight` | `0.6` | Probability weight of the base scenario |
| `scenario_base_pd_multiplier` | `1.0` | PD multiplier applied to `pd_12m` under the base scenario |
| `scenario_upside_weight` | `0.2` | Probability weight of the upside scenario |
| `scenario_upside_pd_multiplier` | `0.8` | PD multiplier applied to `pd_12m` under the upside scenario |
| `scenario_downside_weight` | `0.2` | Probability weight of the downside scenario |
| `scenario_downside_pd_multiplier` | `1.5` | PD multiplier applied to `pd_12m` under the downside scenario |
| `staging_basis` | `base_case` | `base_case` or `scenario_weighted` - which PD basis is used for the SICR (stage 2) test |

The three scenario weights must sum to 1.0, otherwise the API returns a 422.

### CSV Format

Custom portfolio CSVs must include the following columns:

```
loan_id, product_type, exposure_at_default, lgd, pd_12m, pd_origination, days_past_due, remaining_term_months, eir
```

`product_type` can be any text label for your portfolio segment (e.g.
`mortgage`, `sme_term_loans`). Since ECL is normally calculated separately
per portfolio, a single CSV upload would typically contain one
`product_type` value, though the dashboard will happily filter and display
a mix if you give it one.

## Deployment

The app deploys to AWS with no custom domain - CloudFront serves the
frontend, and the FastAPI backend runs on Lambda (as a container image,
for pandas compatibility) behind an API Gateway HTTP API. Everything is
defined in Terraform under [`infra/`](infra/), and a GitHub Actions
workflow ([`.github/workflows/ci-cd.yml`](.github/workflows/ci-cd.yml)) runs
the backend tests and frontend build on every push/PR, then - if those pass -
builds the backend image, applies the Terraform config, and publishes the
frontend to S3/CloudFront. The deploy job runs automatically on every push
to `main`, or on demand via the Actions tab.

> **Live demo**: https://https://drmymq1mrcj43.cloudfront.net
